Slide 1 — Chapter 12 — Emerging trends

The chapter closes by looking ahead. This final part surveys serverless compute over object storage, graph and time-series specialty stores, and high-throughput messaging platforms that shape the next generation of scalable pipelines.

Slide 2 — Learning objectives

By the end of this part, learners should be able to explain serverless compute paired with durable object storage; position Neo4j for relationship-heavy graphs and InfluxDB for timestamped metrics; describe Kafka as a high-throughput messaging backbone for live analytics; and connect these trends to the storage foundations covered earlier in the chapter.

Slide 3 — Directions still shifting operations

Section 12.2 already covered cloud object stores, SQL versus NoSQL trade-offs, and CAP-style consistency. This closing section highlights directions that are still shifting how teams operate those foundations: serverless compute over object storage, specialized database engines, and automated pipelines that keep storage and analytics synchronized.

Slide 4 — Example 12.26 — Serverless Compute Over Object Storage

Beyond managed buckets themselves, teams often attach event-driven functions so compute scales independently of durable storage capacity. Example 12.26 pairs S3-class storage with serverless compute: object stores hold durable blobs while serverless functions process events on ingest, scaling compute independently of storage capacity. The example 26 module for this chapter summarizes that separation of concerns. Cloud-native databases such as Aurora and Spanner further reduce operational burden with automatic scaling for transactional workloads.

Slide 5 — Example 12.27 — Neo4j for Relationship-Heavy Graphs

Horizontal NoSQL stores remain important; the emerging pattern is purpose-built engines for relationship and time-series workloads, often as database-as-a-service. Example 12.27 places graph databases in network-style apps. Graph databases such as Neo4j fit social graphs and fraud rings where multi-hop relationships are the primary query pattern. The example 27 module for this chapter covers that graph fit.

Slide 6 — Example 12.28 — InfluxDB for Timestamped Metrics

Example 12.28 places time-series stores in monitoring workloads. Time-series databases such as InfluxDB store high-rate sensor and market ticks for real-time monitoring and rollups. Purpose-built engines avoid forcing every metric into a general document or relational schema. The example 28 module for this chapter frames that time-series role.

Slide 7 — Example 12.29 — Kafka High-Throughput Messaging

Chapter 9 covers streaming collection; here the operational concern is wiring durable buses so storage and analytics stay synchronized. Example 12.29 returns to Kafka in the pipeline-automation context: Kafka sustains high message rates across producers and consumers, making it a common backbone for fraud detection and live analytics pipelines. The example 29 module for this chapter summarizes that messaging role.

Slide 8 — Looking ahead to Chapter 13

Scalable management provides the storage, consistency, versioning, and lineage substrate for production datasets. Chapter 13 turns to reproducibility and open science: pinning analyses to the versions and provenance records that platforms like those in this chapter must expose. Emerging tools matter only when they still expose addressable versions and auditable lineage.

Slide 9 — Takeaways

Serverless compute scales independently of durable object storage. Neo4j and InfluxDB illustrate purpose-built graph and time-series engines. Kafka remains a high-throughput messaging backbone for live pipelines. Trends build on earlier foundations rather than replacing CAP, formats, versioning, and lineage. Chapter 13 consumes the managed assets this chapter prepares.
